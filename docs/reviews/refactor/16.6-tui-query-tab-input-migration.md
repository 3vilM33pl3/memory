# TUI — Query Tab Input Migration

## Review Basis

Plan `16.5-tui-app-slim-and-mvu-seed.md` was re-reviewed on 2026-05-19. Nine of the ten tab `update` functions handle real input; `crates/mem-cli/src/tui/tabs/query.rs:444` is still a stub that returns `TabAction::None`:

```rust
pub(in crate::tui) fn update(
    _event: &crossterm::event::Event,
    _state: &mut QueryTabState,
    _ctx: &mut TabContext,
) -> TabAction {
    TabAction::None
}
```

The query tab's input still lives centrally in `crates/mem-cli/src/tui/app.rs`. Plan 16.5 acknowledged this — it ordered `query` last as the "largest input surface" — and accepted the stub as plan-complete because the seam was in place. This plan closes the remaining migration.

## Goal

Move the input handling that can live with the query tab into `tabs/query.rs::update`, without changing keybindings, modal behavior, or the async query lifecycle. Leave global text-input modality (`InputMode::Query`) and background-event dispatch where they are — those are coordinator concerns.

## Scope

In scope (move into `tabs/query.rs::update`):
- Result-list navigation when `active_tab == TabKind::Query` and `input_mode == InputMode::Normal`: `Down`/`j` and `Up`/`k` advance/retreat `query_selected_index` (`app.rs:1457`, `:1460`).
- Result-list selection state: keep `query_table_state` in sync with `query_selected_index`.
- Detail-pane open trigger (whatever key currently opens the selected memory's detail — locate the call site at the same time).
- Query-history cursor movement *inside text-input mode* (`app.rs:1913`, `:1917`) — see "deferred" below; this may or may not migrate depending on how clean the extraction is.

Out of scope (stay in `app.rs`):
- Modal entry: `Enter` on the Query tab starting `InputMode::Query` (`app.rs:1731-1742`, `start_query_input` at `:1937`). Modality is a global concern.
- `handle_text_input` framework (`app.rs:1882-1929`). The `TextInputKind::Query` branches mutate `app.query.query_text`, dispatch the query, etc. — these touch the runtime and the background-task channel, which belong on `App`.
- `BackgroundEvent::QueryCompleted` / `QueryDetailLoaded` handling (`app.rs:1227`, `:1236`). Async results dispatch back through the runtime; they stay coordinator-owned.

If `TextInputKind::Query` history navigation (Up/Down inside the input buffer) can be expressed as a `QueryTabState` mutation without touching modality, migrate it. Otherwise leave it and note why in the PR body.

## PR Shape

One PR. Three commits at most:

1. **Move normal-mode navigation**: cut the two `KeyCode::Down|Char('j')` and `KeyCode::Up|Char('k')` arms from `app.rs:1457-1462` (and any adjacent query-only nav like `PageDown`/`Home`/`End` if present); paste into `query::update`. Convert from `self.query.query_selected_index` mutations to `state.query_selected_index`. Return `TabAction::Redraw` when the index changes, `TabAction::None` otherwise.
2. **Move detail-open trigger** (locate before writing the PR): same pattern.
3. **Optional**: history-cursor migration if it's clean.

## Implementation Notes

- `dispatch_update` (`tabs/mod.rs:55`) already runs the tab's `update` *before* global handling. To preserve current behavior, the tab must return `TabAction::Redraw` or `TabAction::None` — never `TabAction::Quit` or `SwitchTab` for these keys.
- Selection-bound side effects (e.g. clearing `query_selected_detail` on index change) are pure `QueryTabState` mutations — keep them inside `update`.
- The `dispatch_update` placeholder `let _ = (&ctx.project, ctx.active_tab); let _ = (TabAction::Redraw, ...)` block in `tabs/mod.rs:62-67` exists to suppress unused-var warnings while some tabs are stubs. Once `query::update` returns non-`None` values, audit whether this placeholder block can be removed.
- Do not introduce any new keybinding. Do not change what `Enter` does on the Query tab.
- If a navigation key needs to reach `TextInputKind::Query` history (i.e. the key has different meaning in normal vs. input mode), the tab's `update` only handles the normal-mode case; the input-mode case stays in `handle_text_input`.

- `crates/mem-cli/src/tui/tabs/query.rs::update` handles at least the two navigation keys (`Down`/`j`, `Up`/`k`) and returns `TabAction::Redraw` on selection change; the corresponding arms are gone from `app.rs`.
- `grep -n "active_tab == TabKind::Query" crates/mem-cli/src/tui/app.rs` returns no normal-mode-only navigation matches. Modal-entry and background-event matches are acceptable.

- At least three unit tests cover query-tab navigation under the new `update`.
- No visible TUI behavior changes. Keybindings, modal entry, query lifecycle, detail-pane behavior all unchanged.
