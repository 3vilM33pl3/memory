# TUI — Slim `App` and Seed MVU Update Seam

## Review Basis

Plan `16-tui-architecture-split-finish.md` was re-reviewed on 2026-05-19. Two of its four acceptance criteria did not land:

- `struct App` (`crates/mem-cli/src/tui/state.rs:24`) still has **34 direct fields** vs. the ~10 target. The 10 per-tab sub-state fields landed cleanly; the other 24 coordinator/global fields stayed flat on `App`.
- The per-tab MVU seam the plan called for did not land. Tab modules expose only `pub fn draw_<name>_tab(frame, app: &App, area)` (e.g. `crates/mem-cli/src/tui/tabs/errors.rs:10`) — they take the *full* `App`, not just their sub-state, and there is no `update(event, &mut state, &ctx) -> Option<Action>` counterpart. Event handling still lives centrally in `crates/mem-cli/src/tui/app.rs`.

Side observation: `crates/mem-cli/src/tui/render.rs` is now 3,980 LOC — the new heaviest file in the crate. Plan 16 did not budget it, but the pattern (god-file moved rather than dissolved) is the same one this pack is closing.

The structural pieces of plan 16 (tabs locatable, `tui.rs` collapsed to 9 LOC, per-tab state structs in `state.rs`, clippy clean) are accepted as shipped. This plan picks up the slim-`App` and MVU-seed items only.

## Goal

Get `struct App` to a coordinator shape (~10 fields) and introduce a per-tab `update` seam so tab modules own both render *and* event handling for their own state. Once tabs own their input, future refactors can touch one tab without reading the whole coordinator.

## PR Shape

Move-only PRs where possible. Each PR should be reviewable as a `git diff -M` largely-moves diff.

1. **Group coordinator fields into sub-structs**. New types in `crates/mem-cli/src/tui/state.rs`:
   - `struct ServiceState { health_ok, backend_connection_state, service_role, service_health_state, service_database_state, manager_status, restart_notice, stream_connecting, relay_discovery_enabled }`
   - `struct UiChrome { help, ui_status, status_message, input_mode, needs_redraw }`
   - `struct RuntimeMeta { overview, versions, skill_inventory, startup_at, profile }`
   - `App` keeps: `project`, `repo_root`, `active_tab`, the 10 per-tab sub-states, `filters`, `background_tx`, `service: ServiceState`, `chrome: UiChrome`, `meta: RuntimeMeta`. Target: ≤ 18 direct fields, ideally ~15.
   - All field reads through `app.service.health_ok` etc. — pure rename. No behavior change.

2. **Introduce `TabUpdate` trait and per-tab `update` fns**. In `crates/mem-cli/src/tui/tabs/mod.rs`:
   ```rust
   pub(in crate::tui) enum TabAction {
       None,
       Redraw,
       SwitchTab(TabKind),
       Quit,
       /* domain actions tabs need to surface upward */
   }
   pub(in crate::tui) struct TabContext<'a> { /* shared read-only state tabs need */ }
   ```
   Each tab module gains:
   ```rust
   pub(in crate::tui) fn update(
       event: &crossterm::event::Event,
       state: &mut <Tab>State,
       ctx: &mut TabContext<'_>,
   ) -> TabAction;
   ```
   The central event loop in `app.rs` dispatches to `tabs::<active>::update(...)` before falling back to global keymap handling. Tabs that don't yet have moveable input return `TabAction::None`.

3. **Per-tab event migration** (one PR per tab, in this order — smallest input surfaces first):
   - `errors`, `embeddings`, `watchers`, `agents` (mostly navigation keys)
   - `activity`, `project`, `resume`, `review`
   - `memories`, `query` (largest input surfaces — text input, history scroll, detail panes)
   Each PR moves the matching key-handling block out of `app.rs` and into the tab's `update`. No new behavior, no new keys.

4. **Optional follow-up — `render.rs` dissolution**. If time permits, move per-tab render plumbing out of `crates/mem-cli/src/tui/render.rs` (3,980 LOC) into the matching `tabs/<name>.rs`. Cross-tab chrome (footer, header, modal overlays) stays in `render.rs` or moves to `tui/chrome.rs`. Out of scope by default — call out explicitly if attempted.

## Implementation Notes

- Field grouping in PR 1 is a pure rename: every `self.health_ok` becomes `self.service.health_ok`. Use the IDE's rename refactor where possible; `cargo check` after each grouping.
- `TabContext` should expose only what tabs legitimately need (project slug, profile, background_tx, filters). It must not hand over `&mut App` — that defeats the point.
- Tabs that surface domain events (e.g. "memory selected → fetch detail") return a `TabAction::FetchMemoryDetail(uuid)`-style variant; the central loop translates that into the existing background-task call. Do not let tabs spawn tasks directly.
- Keep input mode (modal vs. normal) handling central in `app.rs` for now. Tabs only see events when the input mode allows tab-local handling.
- Do not change rendered output, keybindings, tab order, or any externally observable behavior.
- If a tab's existing key block in `app.rs` mutates non-tab state (e.g. `app.status_message`), that mutation either moves into the `TabAction` payload or stays in `app.rs` after the tab's `update` runs. Document which.

## Tests

- `cargo test -p mem-cli --all-targets --locked` after each PR.
- `cargo clippy -p mem-cli --all-targets --locked -- -D warnings`.
- Add a unit test per tab in `crates/mem-cli/src/tui/app/tests.rs` (or new `tabs/<name>_tests.rs`) covering one representative key: e.g. `errors_tab_down_arrow_advances_selection`. Cheap once tabs own their state.
- Manual smoke: open the TUI, cycle every tab, exercise one keybinding per tab, confirm the footer/status/help overlays still behave.

## Acceptance Criteria

- **`struct App` field budget**: `crates/mem-cli/src/tui/state.rs` — `struct App` has ≤ **18** direct fields, with `service: ServiceState`, `chrome: UiChrome`, `meta: RuntimeMeta` (or equivalent grouping) absorbing the rest.
- **MVU seam present**: every tab module in `crates/mem-cli/src/tui/tabs/` exposes a `pub(in crate::tui) fn update(event, &mut <Tab>State, &mut TabContext<'_>) -> TabAction`.
- **Central event loop dispatches through tabs**: `crates/mem-cli/src/tui/app.rs` calls `tabs::<active>::update(...)` for the active tab before any global handling. Verifiable by `grep -n "tabs::.*::update" crates/mem-cli/src/tui/app.rs` returning at least one hit per tab (likely via a `match active_tab` block).
- **Tab signature no longer takes `&App`**: tab `draw_*` and `update` fns take their own sub-state plus a `TabContext` / `&App` read-only reference *only where necessary for chrome*. `grep -n "fn draw_.*&App\b" crates/mem-cli/src/tui/tabs/*.rs` should trend toward zero; ≤ 2 remaining hits acceptable for tabs that genuinely need global state for layout.
- No visible TUI behavior changes. Tab order, keybindings, rendered output, footer/help/status all unchanged.
