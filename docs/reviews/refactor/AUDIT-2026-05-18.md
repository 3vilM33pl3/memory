## 4. Pattern Analysis

Three observations explain why so many structural plans landed cosmetically:

1. **"Minimum viable extraction" pattern.** Each structural plan landed exactly one representative extraction (one command family, one router function, one tab, one helper file, one repository module) — enough to prove the pattern compiles, not enough to discharge the goal. The plans explicitly invited per-PR splits ("Use several small PRs, each moving one command family") but the follow-up PRs never came.

2. **God-files grew during the refactor window.** `mem-cli/main.rs` (+1,008 LOC), `mem-service/main.rs` (+259 LOC), and several others got *bigger* because the same release window also shipped `memory status`, the provenance verifier, and the web-parity work — and those features were written into the existing god-files instead of into the new module scaffolds. The split's gains were eaten by concurrent feature growth.

3. **Acceptance criteria were unmeasured.** The plans had numeric-flavor acceptance criteria ("main.rs loses meaningful size", "a reviewer can locate auth without searching an 8k-line file") but no quantitative target. With no LOC budget, "we extracted one module" passed self-review.
