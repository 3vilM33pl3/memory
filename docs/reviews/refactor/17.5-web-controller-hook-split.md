# Web — Controller Hook Split

## Review Basis

Plan `17-web-app-decomposition-finish.md` was re-reviewed on 2026-05-19. All 11 tabs landed under `web/src/features/<name>/`, `App.tsx` collapsed to 345 LOC, tests pass — acceptance criteria met. The decomposition pushed cross-cutting state into a new file: `web/src/hooks/useWebAppController.ts`, currently **1,067 LOC**.

`useWebAppController` is now the heaviest file in `web/`:
- ~30 `useState` hooks covering memories, query, agents, activities, bundles, embeddings, errors, resume, review, watchers, plus global app-shell state (`web/src/hooks/useWebAppController.ts:125-192`).
- 16+ `useEffect` blocks orchestrating fetch lifecycles.
- A single returned object with ~80 fields that `App.tsx` destructures and fans out as per-tab prop bundles.

This is the same pattern the audit flagged for `App.tsx` before plan 17 — a god-file. Plan 17 did not budget hooks, so the work landed; this follow-up addresses the residual.

## Goal

Slice `useWebAppController` into per-feature controller hooks co-located with each tab, plus a thin `useAppShell` for genuinely cross-cutting state. Each tab owns its data fetching, mutation, and local UI state.

## Scope

In scope (becomes a per-feature hook next to the tab component):
- Memories — list, filters, selection, history, delete
- Query — input, history cursor, results, selected detail
- Agents — snapshot, selection
- Activity — events, selected activity
- Bundles — export/import options + previews
- Embeddings — backends, selection, operations
- Errors — diagnostics aggregation
- Project (tab) — runtime status, LLM audit
- Resume — resume data, autoload
- Review — proposals, replacement policy
- Watchers — already minimal; may stay prop-driven

In scope (stays cross-cutting, in `useAppShell`):
- Active tab + tab switching
- Project selection (`project`, `projectInput`, `repoRootInput`, `applyProjectInput`)
- Connection state, health, service version
- Overview response (used by multiple tabs as identity/context)
- Help-panel open state
- Search/query input refs that are focused by global keybindings

Out of scope:
- API client (`web/src/api.ts`) layout — unchanged.
- Visible behavior, routes, styling — unchanged.
- Type definitions in `web/src/types.ts` — unchanged unless a type is genuinely feature-local.
- Drive-by: the `MemoryTypeFilter`/`StatusFilter` re-export from `features/memories/MemoriesTab` referenced by the central controller can move to `features/memories/types.ts` in the same PR as the memories slice.

## PR Shape

One PR per feature. Move-only style — each PR moves one feature's `useState`/`useEffect`/`useCallback` block out of `useWebAppController` into a new `features/<name>/use<Name>Controller.ts`, and updates the tab component to call its own hook instead of receiving props. `App.tsx` shrinks one tab at a time.

Suggested order (leaf-shaped state first, deeply entangled tabs last):

1. `features/watchers/useWatchersController.ts` (smallest if any state — may collapse to just prop forwarding)
2. `features/errors/useErrorsController.ts`
3. `features/agents/useAgentsController.ts`
4. `features/embeddings/useEmbeddingsController.ts`
5. `features/bundles/useBundlesController.ts`
6. `features/resume/useResumeController.ts`
7. `features/project/useProjectTabController.ts` (runtime status + LLM audit; not the project-selection chrome)
8. `features/review/useReviewController.ts`
9. `features/activity/useActivityController.ts`
10. `features/memories/useMemoriesController.ts`
11. `features/query/useQueryController.ts` (largest entanglement: history cache, roundtrip timing, citation refs)

Then a final PR:

12. **App shell**: rename what's left of `useWebAppController` to `useAppShell` (or split into `hooks/useAppShell.ts`). It should hold only the cross-cutting state listed in Scope above.

## Implementation Notes

- Each feature hook accepts a `{ project, /* other cross-cutting deps */ }` argument and returns the props the tab needs. The tab calls it once at the top.
- Cross-feature dependencies surface as hook inputs, not as shared mutable state. E.g. `useMemoriesController({ project })` rather than reading from a shared context.
- If two features need the same fetched data (e.g. `overview`), that data lives in `useAppShell` and is passed into the feature hooks as input. No new contexts unless ≥ 3 features need the same data.
- Keep `useCallback`/`useMemo` boundaries identical to today so re-render behavior doesn't drift.
- Effects stay co-located with the state they orchestrate. Don't pull cleanup or fetch logic up into `useAppShell` unless it genuinely coordinates across features.
- Strict TypeScript stays strict. No `any` introduced during moves. Existing typed shapes carry over verbatim.
- Test files stay where they are. Per-tab `*.test.tsx` should keep working without changes; if a test was importing private state from `useWebAppController`, update the import to the new hook.

## Tests

- `npm test` (vitest) in `web/` after each PR — currently 11 files, 11 tests passing.
- For each new feature hook, add at least one render-with-hook smoke test in the existing `<Tab>Tab.test.tsx`. Don't add separate hook tests unless the hook has nontrivial pure logic worth isolating.
- Manual smoke after the final PR: cycle every tab in dev, confirm no fetch storms, no missing data, no console errors.

## Acceptance Criteria

- **`useWebAppController.ts` removed or renamed to `useAppShell.ts`** under `web/src/hooks/`.
- **LOC budget**: the remaining `useAppShell.ts` ≤ **300 LOC**, holding only cross-cutting state from the Scope list above.
- Every tab under `web/src/features/<name>/` has a sibling `use<Name>Controller.ts` (or its tab component is prop-driven from `useAppShell` directly, justified in PR body).
- `App.tsx` ≤ 400 LOC and contains no per-feature state — only `useAppShell` plus tab routing.
- `grep -n "useWebAppController" web/src/` returns no hits after the final PR.
- `grep -nE "^import .*from \"\.\./hooks/useWebAppController\"" web/src/features/` returns no hits.
- No `any` introduced. `npm test` passes (11+ tests, count grows by the number of new hook-render tests).
- No visible UI changes, no fetch-cadence changes, no console errors.
