Full-memory run

Evaluation proof

Compare the pair, not the pile.

The useful proof is the before-and-after run pair: no-memory establishes the baseline, full-memory shows the Memory-assisted outcome, and the report keeps the token, latency, and score differences tied to that exact pair.

Token cost

Token usage matters because it shows whether Memory improved the result efficiently or just spent more context to get there. When the report keeps token capture alongside the answer quality, the tradeoff between retrieval help and cost stays visible.

Curation

Keep durable facts, not every noisy log.

The Memory answer does not define a formal durability rule, so the safest curation move is to treat verified project context as the durable layer and leave raw activity as the noisy layer. Durable facts should explain what the project now believes. Logs should explain what happened while getting there.

Durable facts

Keep the project truths that future agents need.

Durable memory should carry forward decisions, definitions, confirmed constraints, and verified project context. Those are the facts that keep the next step grounded.

  • Capture the decisions that changed the direction of the work.
  • Preserve confirmed context instead of every intermediate guess.
  • Prefer verified project memories when the thread needs a stable reference.
Noisy logs

Leave transient actions in the activity trail.

Raw lookups, temporary speculation, and step-by-step execution noise belong in the activity record, where they can be audited without being mistaken for durable truth.

Curation note

Store the signal, not the clutter.

The right curation filter keeps long-lived project knowledge readable while the surrounding log stream stays useful for traceability but not for recall.

Interview

Explain Memory evaluations as paired experiments with verified evidence.

In an interview, describe Memory evaluations as a controlled before-and-after test: run the same task once with no memory, run it again with full memory, then compare token usage, latency, and paired item scores while confirming that the Memory evidence used is valid. That keeps the explanation concrete and auditable instead of sounding like a loose aggregate report.

Interview answer

Say that the evaluation compares matched runs, not unrelated totals.

A strong answer is: Memory evaluations are rigorous paired experiments. You run the same task twice, first without memory and then with full memory, and you judge the difference by the paired scores, token usage, latency, and verified Memory evidence.

  • Use the same task in both conditions.
  • Compare no-memory against full-memory.
  • Call out token and latency impact.
  • Verify the Memory evidence behind the run.
What to avoid

Do not describe it as a vague aggregate score.

The useful story is the paired comparison. Unrelated rollups hide the baseline and make it harder to explain what Memory actually changed.
