Activity ledger

Log token-tracked actions so every step has a visible cost and outcome.

The ledger records each action, the tokens it consumed, and the result it produced so operators can review usage without guessing where the budget went.

What gets tracked

Each significant action writes a compact entry with the actor, the token count, and the reason the work was allowed to continue. That keeps the audit trail readable while still showing the operational cost.

  1. Reserve tokens

    Estimate the cost before execution starts so the system can stop early if the budget is too small.

  2. Record actions

    Capture prompts, retrievals, and handoffs as ledger entries with a clear token total attached.

  3. Review usage

    Compare the stored totals against the outcome to spot expensive steps that did not add enough value.

  4. Carry forward context

    Pass the ledger history into the next task so the team can see what already spent tokens and why.

Token cost

Capture tokens early so the team can see the real cost before it compounds.

Token capture turns invisible spend into a visible signal. When every prompt, retrieval, and handoff carries a cost, the team can compare approaches, spot waste, and keep the memory loop efficient instead of guessing where the budget went.

Why capture matters

Recorded token totals make tradeoffs concrete. A short answer, a broader retrieval, or an extra agent hop can all look reasonable in isolation, but the ledger shows which path actually stays within budget.

What the team learns

With visible costs, operators can prune expensive steps, defend the right level of context, and keep future runs aligned with the same budget target.

Memory types

Use the right memory shape for the job instead of forcing every note into one bucket.

Plan, implementation, debugging, reference, and project memories each answer a different question. Separating them keeps retrieval precise, helps the team know what to trust, and makes the stored context easier to maintain over time.

Plan memory

Store intent, milestones, sequencing, and decision criteria so the team can see what is supposed to happen next.

Implementation memory

Capture how the system was built, including important architecture choices, code paths, and integration details.

Debugging memory

Record symptoms, root cause, and fix so the same issue can be resolved faster if it appears again.

Reference memory

Keep durable facts, links, definitions, and background material that the team may need to consult repeatedly.

Project memory

Preserve the broader context for scope, ownership, status, and the decisions that shape the work across the whole project.

Curation

Keep durable facts and filter out noisy logs before they crowd the signal.

Good curation keeps memory useful over time. Durable facts belong in the record because they describe decisions, definitions, ownership, or other context the team will need again. Noisy logs document activity, but they are often too repetitive, transient, or low-value to preserve as memory.

Durable facts

Keep statements that stay true across many sessions: final decisions, stable constraints, key project context, and reference details that help the next person reason correctly without re-deriving the answer.

Noisy logs

Treat repetitive status updates, low-signal traces, and short-lived execution chatter as logs instead of memory. They are useful for debugging in the moment, but they usually should not outlive the task that produced them.

Simple rule

Ask whether the note will still matter after the current run. If it explains a durable decision or a recurring fact, curate it into memory. If it only reflects momentary activity, keep it as operational noise.

Result

This split keeps retrieval clean, reduces drift from irrelevant details, and makes the memory layer easier to trust when the team needs a grounded answer fast.

Local first, privacy by default

Keep sensitive context close to the user and limit what leaves the device.

Local-first memory keeps the smallest necessary data path in play. Store and process what you can on the machine, sync only what is needed, and make every transfer explicit so privacy stays easy to understand and audit.

Local storage

Keep working notes, cached context, and short-lived state on the device so the core loop stays fast without exposing more data than the task requires.

Minimal sharing

Send only the fields that are needed for retrieval or collaboration and avoid copying full histories when a summary will do the job.

Consent and control

Make retention, sync, and deletion choices visible so people can decide what stays local, what moves upstream, and what should be removed entirely.

Clear boundaries

Separate private context from shared project memory so the system can respect confidentiality while still giving the team the facts they need.

Architecture strip

The service coordinates requests, Postgres/pgvector stores retrieval state, the graph layer preserves relationship context, and embeddings connect the query to the right semantic neighborhood.

Graph

Embeddings

Interview proof

Describe memory evaluations as a before-and-after test, not a vague promise.

In interviews, explain that the evaluation starts with no-memory so the baseline is honest, then compares the same prompt with retrieval enabled to show whether memory actually improves the answer.

Simple script

"We run the same task twice. First with no stored memory, then with memory enabled. If the second run finds the right note faster, stays aligned, and avoids re-deriving context, the evaluation passes."

What to emphasize

  • Start from a clean baseline so the comparison is fair.
  • Measure recall speed, alignment, and whether the answer uses stored context.
  • Describe results as observable behaviors, not platform claims.
